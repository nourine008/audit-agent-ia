# OWASP A07 Identification and Authentication Failures

## Description
Cette vulnérabilité est présente lorsqu'un attaquant parvient à tromper un système pour qu'il reconnaisse un utilisateur illégitime comme authentifié. Elle découle de la faiblesse des contrôles liés à la gestion des identités, des mots de passe ou des sessions actives.

L'application est vulnérable si :
1. Permet les attaques automatisées : Absence de protection contre le Credential Stuffing (utilisation de listes de mots de passe fuités), le Password Spraying (tester des variations comme `Password1!`, `Password2!`) ou la force brute.
2. Autorise les mots de passe faibles : Acceptation de mots de passe par défaut (`admin`/`admin`) ou ultra-communs.
3. Gestion des sessions non sécurisée : Exposition de l'identifiant de session dans l'URL, réutilisation du même identifiant après la connexion, ou absence d'invalidation (Timeout) après déconnexion ou inactivité.
4. Défaillance du Single Sign-On (SSO) :Le fait de se déconnecter d'une application ne ferme pas la session globale (absence de Single Logout), laissant les autres applications accessibles.
5. Absence ou contournement du MFA :Pas d'authentification multifacteur ou présence de solutions de secours (fallbacks) non sécurisées.

## Comment corriger cette faille
1.Implémenter le MFA (Multi-Factor Authentication) :Rendre obligatoire le second facteur pour bloquer les attaques automatisées de réutilisation d'identifiants.
2. Interdire les mots de passe compromis : Valider les nouveaux mots de passe par rapport à des listes de dictionnaires (ex: les 10 000 plus mauvais mots de passe) ou des API comme *Have I Been Pwned*.
3. Durcir les retours d'erreurs (Anti-Énumération) :*Utiliser des messages génériques et identiques pour tous les scénarios d'échec (ex: « Identifiant ou mot de passe incorrect ») afin d'empêcher un attaquant de deviner les noms d'utilisateurs valides.
4. Limiter les tentatives (Rate Limiting) : Bloquer temporairement ou ralentir progressivement les requêtes après plusieurs échecs de connexion, sans pour autant créer un déni de service (DoS).
5. Gestion stricte des identifiants de session : Générer un nouvel identifiant de session hautement aléatoire immédiatement après la connexion, le stocker exclusivement dans un cookie sécurisé (`HttpOnly`, `Secure`, `SameSite`), et détruire impérativement la session côté serveur lors de la déconnexion.

## Exemple de Code Vulnérable (Scenario #1 & #3 - Authentification et Session en PHP)
```php
// VULNÉRABILITÉ 1 : Message d'erreur trop précis permettant l'énumération de comptes
$username = $_POST['username'];
$password = $_POST['password'];

$stmt = $db->prepare("SELECT * FROM users WHERE username = ?");
$stmt->execute([$username]);
$user = $stmt->fetch();

if (!$user) {
    die("L'utilisateur n'existe pas."); // L'attaquant sait maintenant que ce nom d'utilisateur est libre
} else if ($user['password'] !== sha1($password)) { // Utilisation d'un hash faible (sha1)
    die("Mot de passe incorrect pour cet utilisateur."); 
}

// VULNÉRABILITÉ 2 : Déconnexion incomplète (Uniquement côté client)
// Le cookie est effacé dans le navigateur, mais la session reste active et valide sur le serveur !
setcookie("session_id", "", time() - 3600);


##Exemple de code sécurisé (php)

session_start();

$username = $_POST['username'];
$password = $_POST['password'];

// 1. Limitation des tentatives (Rate Limiting imagé ici via une fonction globale)
if (is_ip_brute_forcing($_SERVER['REMOTE_ADDR'])) {
    http_response_code(429);
    die("Trop de tentatives. Veuillez réessayer plus tard.");
}

$stmt = $db->prepare("SELECT * FROM users WHERE username = ?");
$stmt->execute([$username]);
$user = $stmt->fetch();

// CORRECTION 1 : Message d'erreur identique et générique (Bloque l'énumération)
// CORRECTION 2 : Utilisation de password_verify pour valider le hash fort (bcrypt/Argon2)
if ($user && password_verify($password, $user['password_hash'])) {
    
    // CORRECTION 3 : Régénération immédiate de l'ID de session après connexion (Prévient la fixation de session)
    session_regenerate_id(true);
    
    $_SESSION['user_id'] = $user['id'];
    $_SESSION['logged_in'] = true;
    echo "Connexion réussie.";
} else {
    // Même message si l'utilisateur n'existe pas ou si le mot de passe est faux
    http_response_code(401);
    die("Identifiant ou mot de passe incorrect.");
}

// --- SCRIPT DE DÉCONNEXION SÉCURISÉ (logout.php) ---
// CORRECTION 4 : Invalidation totale de la session côté serveur ET destruction du cookie
$_SESSION = array(); // Écrase toutes les variables de session
if (ini_get("session.use_cookies")) {
    $params = session_get_cookie_params();
    setcookie(session_name(), '', time() - 42000,
        $params["path"], $params["domain"],
        $params["secure"], $params["httponly"]
    );
}
session_destroy();
